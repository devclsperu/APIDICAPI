Usage Guide - APIDICAPI

Technical documentation for private API record query clients.

Información General

Base URL

https://www.apidicapi.com.pe/api/v1/records

Protocol

HTTPS

Authentication

Bearer Token

Format

JSON

Rate Limiting

Configured per endpoint

Authentication

All requests require authentication via Bearer Token in the header:

Authentication Header 
Authorization: Bearer YOUR_API_TOKEN
cURL Example 
curl -H "Authorization: Bearer YOUR_API_TOKEN" \
     https://www.apidicapi.com.pe/api/v1/records/last-hour

Available Endpoints

GET

/api/v1/records/last-hour

Description: Gets all records from the last hour.

Request Example 
curl -H "Authorization: Bearer YOUR_API_TOKEN" \

GET

/api/v1/records/all-day

Description: Gets all records from the current day (00:00:00 to 23:59:59).

Request Example 
curl -H "Authorization: Bearer YOUR_API_TOKEN" \
     https://www.apidicapi.com.pe/api/v1/records/all-day
GET

/api/v1/records/{id}

Description: Gets records from a specific vehicle by its ID.

Parameters:

  • id (path): Vehicle/beacon ID
Request Example 
curl -H "Authorization: Bearer YOUR_API_TOKEN" \
     https://www.apidicapi.com.pe/api/v1/records/123456
GET

/api/v1/records/last/{hours}

Description: Gets records from the last N hours (2-24 hours).

Parameters:

  • hours (path): Number of hours (2-24)
Request Example 
curl -H "Authorization: Bearer YOUR_API_TOKEN" \
     https://www.apidicapi.com.pe/api/v1/records/last/6
GET

/api/v1/records/select-day

Description: Gets all records from a specific date by dividing the query into two parts (morning and afternoon) for better performance.

Parameters:

  • date (query): Date in DD-MM-YYYY format
Request Example 
curl -H "Authorization: Bearer YOUR_API_TOKEN" \
     "https://www.apidicapi.com.pe/api/v1/records/select-day?date=15-01-2024"

HTTP Status Codes

200 OK - Successful request
400 Bad Request - Invalid parameters
401 Unauthorized - Invalid token
404 Not Found - Endpoint not found
429 Too Many Requests - Rate limit exceeded
500 Internal Server Error - Internal error

Error Handling

Common errors and their solutions:

401

Unauthorized 

{
  "success": false,
  "error": "Unauthorized",
  "details": {
    "message": "Invalid or missing authentication token"
  }
}

Solution: Verify that the token is valid and included in the header.

400

Bad Request

{
  "success": false,
  "error": "Invalid parameter",
  "details": {
    "message": "The hours parameter must be a number between 2 and 24",
    "received": "25"
  }
}

Solution: Verify that parameters meet the validations.

429

Too Many Requests 

{
  "success": false,
  "error": "Too many requests",
  "details": {
    "message": "Rate limit exceeded. Try again later."
  }
}

Solution: Implement delays between requests (maximum 10 requests/second).

404

Not Found

{
  "success": false,
  "error": "Endpoint not found",
  "details": {
    "message": "The endpoint GET /api/v1/records/invalid does not exist",
    "availableEndpoints": [
      "GET /api/v1/records/last-hour",
      "GET /api/v1/records/all-day",
      "GET /api/v1/records/:id"
    ]
  }
}

Solution: Verify that the endpoint URL is correct.

Best Practices

Rate Limiting

  • Maximum 10 requests per second
  • Implement delays between consecutive requests
  • Handle 429 errors with exponential retries

Error Handling

  • Always check the success field in responses
  • Implement retries for temporary errors (5xx)
  • Do not retry client errors (4xx)

Response Validation

  • Verify that response.success is true
  • Validate the structure of response.data
  • Handle cases where data is empty
